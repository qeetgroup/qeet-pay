# Contributing to Qeet Pay

Qeet Pay is a Java 21 + Spring Boot modular-monolith payments backend, with an
operator **console** (`apps/console/`), a **`qp` CLI** (`cli/`), a separate
Python **fraud service** (`fraud-svc/`), and client **SDKs** that live in the
sibling repo `../qeet-sdks/` (never inside this repo). This guide covers how to
build and test each piece, the conventions that keep the ledger and migrations
safe, and the branch/PR flow.

Please also read [`CLAUDE.md`](CLAUDE.md) (architecture + gotchas) and
[`SECURITY.md`](SECURITY.md) (security model) before non-trivial changes.

---

## Prerequisites

- **JDK 21+** — the Gradle wrapper (9.6.1) runs on JDK 21–25; bytecode targets 21.
  The prod image is `eclipse-temurin:21-jre`. `java -version` should report 21+.
- **Docker engine** — required for the backend tests (Testcontainers spins a real
  Postgres) and for `make db-up`.
- **Node 22 + pnpm 9** (Corepack) — for the console.
- **Go 1.25+** — for the `qp` CLI.

## Building & testing each component

### Backend (Java / Spring Boot)

```bash
make db-up                 # Postgres :5201 (+ NATS :4222 / Redis :6379) via docker compose
cp .env.example .env       # dev defaults work out of the box
make dev                   # run the API on :4201 (dev profile — boots without Qeet ID)
make build                 # bootJar
make test                  # ./gradlew test — needs a Docker engine (Testcontainers)
make kill                  # free a stuck :4201

# a single test:
./gradlew test --tests 'com.qeetgroup.qeetpay.platform.tenancy.RlsIsolationTest'
```

`make test` is **integration-test-first**: it runs against a real Postgres via
Testcontainers, verifies the Spring Modulith boundaries (`ModularityTests`), and
proves RLS isolation under the `NOSUPERUSER` role (`RlsIsolationTest`).

> **Local Testcontainers workaround (macOS / Docker Desktop only).** A very new
> Docker Desktop engine can 400 Testcontainers because docker-java's API version
> falls outside the engine's supported range. Run:
>
> ```bash
> DOCKER_HOST=unix://$HOME/.docker/run/docker.sock \
>   DOCKER_API_VERSION=1.43 \
>   TESTCONTAINERS_RYUK_DISABLED=true \
>   ./gradlew test
> ```
>
> The build bridges `DOCKER_API_VERSION` → docker-java's `api.version`.
> **CI/Linux needs none of this** (the GitHub `ubuntu-latest` runner ships a
> stock Docker engine; Ryuk stays enabled there) — the `ci.yml` workflow just
> runs `./gradlew build`.

### Console (`apps/console/` — TanStack Start + Vite + React 19)

```bash
cd apps/console
corepack enable            # provides pnpm 9 (matches pnpm-lock.yaml)
pnpm install
pnpm dev                   # dev server on :3201
pnpm typecheck             # tsc --noEmit
pnpm lint                  # eslint
pnpm build                 # production build (also regenerates routeTree.gen.ts)
```

`src/routeTree.gen.ts` is generated by the Vite plugin and **kept in VCS** (like
the sibling consoles). Adding a route: create `src/routes/_app/<slug>.tsx` with
`createFileRoute("/_app/<slug>")` and register it in `src/config/navigation.tsx`.
Money is **paise everywhere** in the console too (`src/lib/money.ts`).

### CLI (`cli/` — Go)

```bash
cd cli
go build ./...
go vet ./...
go test ./...
gofmt -l .                 # must print nothing; run `gofmt -w .` to fix
```

The module is stdlib-only (`github.com/qeetgroup/qeet-pay/cli`). The built binary
(`cli` / `qp`) is gitignored.

### Fraud service (`fraud-svc/` — Python / FastAPI)

Separate service (off by default; `QEETPAY_FRAUD_ENABLED=false` → allow-all). See
its own README for setup; not required to run the backend locally.

### SDKs

Client SDKs are **not** in this repo. They live in
`../qeet-sdks/qeet-pay-{node,go,react}` — make SDK changes there.

## Conventions that matter

These are load-bearing for a payments system — CI and reviewers enforce them.

- **Modules are Spring Modulith bounded contexts.** Each domain is a direct
  sub-package of `com.qeetgroup.qeetpay`, declared with `@ApplicationModule` in
  its **`package-info.java`**. Respect declared `allowedDependencies`;
  `ModularityTests` fails the build on a boundary violation. `platform/` is the
  only OPEN (shared-infra, no-domain-logic) module.
- **Migrations are Flyway, immutable once applied.** SQL lives in
  `src/main/resources/db/migration` named `V{n}__description.sql` and auto-applies
  on startup. **Never edit an applied migration** — add `V{n+1}__….sql`. Each new
  domain module gets its own schema; register it in `application.yml`'s
  `flyway.schemas` list. Current tip is `V35`.
- **Money is integer minor units.** `amount_minor` BIGINT + ISO currency; convert
  with `BigDecimal` (HALF_UP). **Never use floats for money.**
- **The ledger is append-only.** No UPDATE/DELETE on `ledger.*`; corrections are
  offsetting entries. Postings must balance (Σdebits = Σcredits) — enforced by
  `LedgerService.postEntry` and a deferred DB trigger.
- **RLS is the tenancy boundary.** New tenant tables need a `merchant_id` column
  and an RLS policy; verify isolation under the `NOSUPERUSER` role, not the dev
  superuser.
- **External backends use the sandbox-adapter pattern** (`@ConditionalOnMissingBean`,
  like `kyb/`): ship a sandbox default so the stack runs with no live keys, and a
  live adapter enabled by config (`*_ENABLED=true`).
- **Events go through the transactional outbox** (`platform.outbox_event`, written
  in the same transaction) — not direct publishes.

## Branch & PR workflow

- **Branches:** `main` is the release line, `develop` is integration. Cut feature
  branches off `develop`: `feat/<scope>-<short-desc>`, `fix/<scope>-…`,
  `chore/…`, `docs/…`. Open PRs against `develop` (release PRs `develop → main`).
- **Commits:** small and focused; Conventional-Commits style
  (`feat(payments): …`, `fix(ledger): …`) is preferred and makes history/changelogs
  clean.
- **Before you push:** run the relevant component's build/test/lint locally
  (`make test`, `pnpm typecheck && pnpm lint && pnpm build`, `go test ./... && gofmt -l .`).
- **CI must be green.** Path-filtered workflows run per component — `ci` (backend),
  `console`, `cli` — plus `dependency-review` on every PR and `codeql` on `main`.
  Don't merge red.
- **PRs should include:** what changed and why, the module(s)/migration(s) touched,
  test evidence, and any new env keys (add them to `.env.example`, commented/off,
  and update `SECURITY.md` if they touch the security posture).
- **Never commit secrets** or a real `.env`. See [`SECURITY.md`](SECURITY.md).

## Reporting security issues

Do **not** file security problems as public issues/PRs — follow the private
process in [`SECURITY.md`](SECURITY.md).
